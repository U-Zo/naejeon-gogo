## Context

현재 멤버 검색은 `MembersPage.tsx`와 `MatchmakingPage.tsx`에서 각각 인라인으로 `m.name.toLowerCase().includes(search.toLowerCase())`를 사용한다. 이는 단순 substring 매칭으로, 한글 초성 검색을 지원하지 않는다.

프로젝트에는 이미 `es-hangul@2.3.8` 라이브러리가 설치되어 있으며, `josa` 함수를 사용 중이다. 이 라이브러리에는 한글 문자열에서 초성을 추출하는 `getChoseong` 함수가 포함되어 있다.

## Goals / Non-Goals

**Goals:**
- 검색어가 한글 자음(초성)으로만 구성된 경우, 멤버 이름의 초성과 매칭하여 검색 결과를 반환한다
- 기존 substring 검색 동작은 그대로 유지한다
- 두 페이지(멤버 관리, 매치메이킹)에 동일한 검색 로직을 적용한다

**Non-Goals:**
- 서버 사이드 검색 구현 (현재 클라이언트 사이드 필터링 유지)
- 중성/종성 부분 매칭 (예: "가"로 "강"을 찾는 것) — 초성 전용
- 검색 debounce나 성능 최적화 (현재 멤버 수 규모에서 불필요)

## Decisions

### 1. 공유 검색 유틸리티 함수 추출

**결정:** 검색 필터링 로직을 별도 유틸리티 함수로 추출한다.

**근거:** 현재 두 페이지에 동일한 검색 로직이 인라인으로 중복되어 있다. 초성 검색 로직을 추가하면 복잡도가 높아지므로, 재사용 가능한 함수로 분리하는 것이 합리적이다.

**대안:** 각 페이지에 인라인으로 로직을 중복 — 중복 코드 발생으로 기각.

### 2. 초성 판별 방식

**결정:** `es-hangul`의 `canBeChoseong` 함수를 활용하여 검색어의 각 문자가 초성인지 판별한다.

**근거:** 한글 자음 문자(ㄱ-ㅎ)가 초성으로 사용 가능한지 판별하는 함수가 이미 라이브러리에 있다. 검색어의 모든 문자가 초성인 경우에만 초성 매칭 모드를 활성화한다.

**대안:** 정규표현식으로 자음 범위 직접 체크 — 라이브러리가 이미 있으므로 불필요.

### 3. 초성 추출 및 매칭 방식

**결정:** `es-hangul`의 `getChoseong` 함수로 멤버 이름에서 초성을 추출하고, 검색어와 `includes`로 비교한다.

**근거:** `getChoseong("김철수")`는 `"ㄱㅊㅅ"`를 반환한다. 검색어 `"ㅊㅅ"`가 초성 문자열 `"ㄱㅊㅅ"`에 포함되므로 매칭된다. 이는 기존 substring 매칭과 동일한 UX를 초성에도 제공한다.

### 4. 검색 흐름

**결정:** 검색어가 초성으로만 구성된 경우 초성 매칭, 그 외에는 기존 substring 매칭을 사용한다. 두 방식은 OR 조건으로 결합하지 않고, 입력 타입에 따라 분기한다.

**근거:** "ㄱ"을 입력했을 때 substring으로도 "ㄱ"이 포함된 이름을 찾고 초성으로도 매칭하면 결과가 혼란스러울 수 있다. 입력이 순수 초성이면 초성 매칭만, 그 외에는 기존 매칭만 수행하는 것이 직관적이다. 단, 빈 문자열이나 공백만 있는 경우에는 모든 멤버를 표시한다.

## Risks / Trade-offs

- **[혼합 입력]** 초성과 일반 문자가 섞인 입력(예: "ㄱ철")은 초성 매칭이 아닌 기존 substring 매칭으로 처리된다 → 이는 의도된 동작이며, 혼합 매칭까지 지원하면 복잡도가 과도하게 증가한다.
- **[영문 이름]** 영문 이름은 초성 추출 대상이 아니므로 초성 검색에 매칭되지 않는다 → 기존 substring 검색으로 영문 이름 검색 가능하므로 문제 없다.
